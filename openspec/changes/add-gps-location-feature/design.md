## Context

本项目是基于 STM32L431RCT6 的电子工牌，已具备声音检测和 LCD 显示功能。现在需要添加 GPS 定位功能，使设备能够获取并显示实时位置信息。

**硬件配置：**
- MCU: STM32L431RCT6 (STM32L4 系列，低功耗设计)
- GPS 模块：ATGM336H（支持 GPS/北斗/GLONASS/伽利略四模）
- 显示屏：ST7735 LCD (160x128)
- 通信接口：USART2 (PA2-RX, PA3-TX)，波特率 9600

**技术背景：**
- GPS 模块输出 NMEA 0183 协议标准数据格式
- GNRMC 语句是最简定位信息，包含时间、经纬度、速度、航向等
- 数据格式为度分格式 (ddmm.mmmm)，需要转换为十进制度 (dd.dddddd)
- 模块使用双模模式（GN 前缀），后续可切换为单 GPS(GP) 或单北斗 (BD)

**约束条件：**
- STM32L431RCT6 资源有限，需考虑内存和 CPU 占用
- 使用 HAL 库进行开发，保持与现有代码风格一致
- GPS 模块独立工作，STM32 仅需被动接收数据

## Goals / Non-Goals

**Goals:**
- 实现 USART2 串口中断接收 GPS 数据
- 解析 GNRMC 语句，提取 UTC 时间、经纬度、定位状态
- 实现坐标格式转换（度分→十进制度）
- 在 LCD 屏幕上显示定位信息
- 判断定位数据有效性（A/V 标志）
- 支持后续扩展（网络上传、多模式切换）

**Non-Goals:**
- 不实现其他 NMEA 语句解析（如 GGA、GSA 等）
- 不实现 GPS 模块配置命令发送（如波特率修改、更新率设置）
- 不实现地图匹配或路径规划功能
- 本次不实现网络上传功能（留待后续扩展）

## Decisions

### 1. 串口接收方案：中断 + 缓冲区

**决策：** 使用 USART2 中断接收方式，配合环形缓冲区存储 GPS 数据

**理由：**
- GPS 数据为间歇性发送（通常 1Hz 更新率），中断方式效率高
- STM32L4 的 USART2 支持独立中断，不影响其他外设
- 缓冲区可容纳完整 NMEA 语句（最大 200 字节）
- 与例程保持一致，降低开发风险

**替代方案：**
- DMA 接收：适合高速连续数据流，但 GPS 数据速率低（9600bps），中断足够
- 轮询方式：浪费 CPU 资源，不符合低功耗设计理念

### 2. NMEA 解析策略：字符串分割 + 字段提取

**决策：** 使用 strstr() 和 memcpy() 进行语句识别和字段提取

**理由：**
- NMEA 0183 协议格式固定，以逗号分隔字段
- GNRMC 语句结构稳定，前 6 个字段包含所需信息
- 标准库函数高效可靠，代码简单易维护
- 避免引入额外的解析库，减少依赖

**解析流程：**
1. 检测 `$` 起始符，开始缓存数据
2. 检测到 `GNRMC` 标识，确认语句类型
3. 检测到 `\n` 结束符，触发解析
4. 使用 strstr() 定位逗号分隔符
5. 使用 memcpy() 提取各字段
6. 校验和验证（可选，本次暂不实现）

### 3. 坐标转换算法：浮点运算

**决策：** 使用 float 类型进行坐标转换计算

**理由：**
- STM32L431RCT6 支持 FPU，浮点运算效率高
- 转换公式简单：`度 = dd + mm.mmmm / 60`
- float 精度（7 位有效数字）满足定位需求
- 代码直观易懂，便于调试

**转换示例：**
```
纬度：2236.9453 → 22 + (36.9453 / 60) = 22.615755
经度：11408.4790 → 114 + (08.4790 / 60) = 114.141317
```

### 4. 数据显示方案：独立显示函数

**决策：** 创建专用的 GPS 数据显示函数，在主循环中调用

**理由：**
- 与现有 LCD 显示架构保持一致（参考 LCD_DisplayWaiting 等函数）
- 显示逻辑独立于数据接收和解析，便于维护
- 可根据定位状态（有效/无效）显示不同内容
- 方便后续扩展（如添加图标、状态指示）

**显示内容：**
- 定位状态：`GPS: OK` 或 `GPS: NO FIX`
- 纬度：`LAT: 22.615755 N`
- 经度：`LON: 114.141317 E`
- UTC 时间：`UTC: 084852.000`

### 5. 数据结构：结构体封装

**决策：** 使用结构体封装 GPS 接收数据

**理由：**
- 相关数据集中管理，便于传递和维护
- 标志位（isGetData、isParseData、isUsefull）便于状态管理
- 预留扩展空间（可添加高度、速度等字段）
- 与例程保持一致，降低学习成本

**结构体定义：**
```c
typedef struct {
    char GPS_Buffer[80];      // 原始数据缓冲区
    char isGetData;           // 是否获取到数据
    char isParseData;         // 是否解析完成
    char UTCTime[11];         // UTC 时间
    char latitude[11];        // 纬度
    char N_S[2];              // 北纬/南纬
    char longitude[12];       // 经度
    char E_W[2];              // 东经/西经
    char isUsefull;           // 定位是否有效
} GPS_SaveData;
```

### 6. 主循环集成：非阻塞处理

**决策：** 在主循环中调用 GPS 数据处理函数，与现有业务并行

**理由：**
- 不影响现有的声音检测功能
- GPS 数据更新率低（1Hz），主循环处理足够及时
- 非阻塞设计，不会导致界面卡顿
- 符合现有代码架构（参考 SoundDetector_Process）

**主循环流程：**
```c
while (1) {
    LED_Process();
    SoundDetector_Process();
    GPS_Process();  // 新增
    LCD_DisplayUpdate();
}
```

## Risks / Trade-offs

### 风险 1：串口冲突或数据丢失

**风险描述：** USART2 中断可能与其他中断冲突，或在高负载时丢失数据

**缓解措施：**
- USART2 中断优先级设置为中等（高于 LED，低于声音检测）
- 接收缓冲区预留足够空间（200 字节）
- 中断服务程序尽量精简，只做数据缓存

### 风险 2：NMEA 解析失败

**风险描述：** GPS 信号弱时可能收到不完整或错误的数据

**缓解措施：**
- 添加数据有效性校验（定位状态字段 A/V）
- 解析失败时清空缓冲区，等待下一帧
- 显示界面提示"NO FIX"或"搜索中..."

### 风险 3：坐标转换精度损失

**风险描述：** float 类型可能导致精度损失

**缓解措施：**
- float 精度（7 位有效数字）对于经纬度足够（小数点后 5-6 位）
- 如需更高精度，可改用 double（会增加 CPU 开销）
- 实际测试验证精度是否满足需求

### 风险 4：内存占用

**风险描述：** 缓冲区和结构体占用 RAM

**缓解措施：**
- GPS 缓冲区 200 字节，结构体约 130 字节，总计 330 字节
- STM32L431RCT6 有 64KB SRAM，占用可忽略
- 后续可优化为环形缓冲区，减少内存占用

### 风险 5：冷启动时间长

**风险描述：** GPS 模块冷启动可能需要 30 秒以上才能定位

**缓解措施：**
- 显示界面提示"正在搜索卫星..."
- 显示已搜索到的卫星数量（如果 GNRMC 不包含，可解析 GPGSV 语句）
- 使用有源天线或改善安装位置

## Migration Plan

### 实施步骤

1. **硬件准备**
   - 连接 ATGM336H 模块：PA2(RX)、PA3(TX)、VCC(3.3V)、GND
   - 确认电源稳定，天线连接可靠

2. **创建 GPS 模块文件**
   - `Core/Inc/gps.h` - 头文件
   - `Core/Src/gps.c` - 实现文件

3. **配置 USART2**
   - 初始化 PA2、PA3 引脚（复用功能）
   - 配置 USART2 参数（9600-8-N-1）
   - 使能 USART2 中断

4. **集成到主程序**
   - 在 main() 中初始化 GPS 模块
   - 在主循环中调用 GPS 数据处理函数

5. **测试验证**
   - 户外测试，确认能正常接收 GNRMC 数据
   - 验证坐标转换准确性
   - 测试显示效果

### 回滚策略

如需回滚：
1. 删除 `gps.c`、`gps.h` 文件
2. 恢复 `main.c` 到修改前版本
3. 断开 GPS 模块连接

## Open Questions

1. **是否需要支持其他 NMEA 语句？**
   - 当前只解析 GNRMC，后续可能需要 GPGGA（高度信息）、GPGSV（卫星信息）

2. **是否需要配置 GPS 模块参数？**
   - 如更新率（默认 1Hz）、波特率（默认 9600）、输出语句类型

3. **坐标系统一性问题？**
   - GPS 使用 WGS-84 坐标系，百度地图使用 BD-09，高德使用 GCJ-02
   - 如需与地图服务对接，可能需要坐标转换

4. **低功耗考虑？**
   - STM32L4 主打低功耗，GPS 模块是否需要间歇性工作以节省电量

5. **数据上传协议？**
   - 后续联网上传时，使用什么协议（HTTP/MQTT/CoAP）
   - 数据格式（JSON/Protobuf）
