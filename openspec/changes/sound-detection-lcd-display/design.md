## Context

本项目需要在 STM32 平台上实现声音触发的 LCD 切屏显示功能。系统已配置：
- MAX9814 音频放大模块连接到 ADC1_IN1 (PC0)
- TIM2 配置为 16kHz 触发源（PSC=0, ARR=4999, 80MHz 时钟）
- DMA 配置为 Half Word 模式，循环传输
- ST7735 LCD 驱动已就绪

当前需要设计完整的软件架构来实现从音频采集到声音检测再到 LCD 显示的完整链路。

**约束条件**：
- 采样率必须保持 16kHz（由 TIM2 硬件触发保证）
- DMA 缓冲区大小有限（SRAM 限制）
- 实时性要求：声音检测延迟 < 100ms
- 底噪处理：环境噪声约 20-50 ADC 值（需实测调整）

**利益相关者**：
- 终端用户：期望灵敏可靠的声音触发体验
- 开发者：需要清晰的代码结构和可调参数

## Goals / Non-Goals

**Goals:**
- 实现基于 DMA 的连续音频采样，CPU 零干预
- 实现实时声音检测算法，能识别打响指、喊叫等明显声音
- 实现底噪自动补偿和环境噪声适应
- 检测到声音时切换 LCD 显示"Sound Detected"
- 提供防抖逻辑避免重复触发
- 代码结构清晰，参数可调

**Non-Goals:**
- 不支持语音识别或特定声音模式识别
- 不支持多声道或立体声
- 不支持音频录制和存储
- 不实现复杂的 FFT 频谱分析（性能考虑）
- 不实现网络传输或蓝牙通信

## Decisions

### 1. 声音检测算法选择

**决策**：采用**时域幅度检测 + 底噪阈值比较**方案

**方案对比**：
- **方案 A**：时域幅度检测（本方案）
  - 优点：计算简单，响应快，资源占用少
  - 缺点：无法区分频率，可能误触发
- **方案 B**：FFT 频域分析
  - 优点：可识别特定频率（如掌声特征频率）
  - 缺点：计算复杂，需要 DSP 库，延迟高

**选择理由**：
- 打响指和喊叫在时域表现为幅度突变，简单检测即可
- 系统时钟 80MHz，但需预留资源给其他任务
- 底噪 + 阈值方案已足够可靠

### 2. DMA 缓冲区大小

**决策**：使用**双缓冲区策略**，每缓冲区 128 采样点

**配置**：
```c
#define ADC_BUFFER_SIZE 128
uint16_t adc_buffer[ADC_BUFFER_SIZE];
```

**理由**：
- 128 点 @ 16kHz = 8ms 数据，满足实时性
- Half DMA 和 Full DMA 中断各处理 64 点，响应更快
- 内存占用：128 × 2 字节 = 256 字节，可接受
- 太大导致延迟高，太小导致频繁中断

### 3. 底噪处理方式

**决策**：采用**动态底噪跟踪 + 固定阈值偏移**

**算法**：
```c
// 底噪跟踪（低通滤波）
noise_floor = noise_floor * 0.98 + min_signal * 0.02;

// 触发阈值
trigger_threshold = noise_floor + DELTA_THRESHOLD; // DELTA=150~300
```

**理由**：
- 环境噪声会随时间变化（如空调开关）
- 纯固定阈值在不同环境下表现不稳定
- 动态跟踪可自适应环境变化
- 固定偏移保证触发灵敏度

### 4. 信号幅度计算方法

**决策**：使用**峰值 - 峰值（Peak-to-Peak）**算法

**计算**：
```c
// 在 64 点窗口内
max_val = max(adc_samples);
min_val = min(adc_samples);
amplitude = max_val - min_val;

// 考虑直流偏置（约 1.6V @ 3.3V 参考 = 2048 ADC 值）
center_offset = 2048;
```

**理由**：
- MAX9814 输出是交流信号叠加直流偏置
- 峰值 - 峰值直接反映信号波动幅度
- 计算简单，只需比较和减法
- 比 RMS 计算更快

### 5. 防抖逻辑设计

**决策**：使用**触发锁定 + 时间窗口**机制

**逻辑**：
```c
if (amplitude > threshold && !trigger_locked) {
    trigger_event();
    trigger_locked = true;
    start_timer(COOLDOWN_MS); // COOLDOWN=500ms
}

if (timer_expired) {
    trigger_locked = false;
}
```

**理由**：
- 声音持续期间会连续触发，需锁定
- 500ms 冷却期避免重复显示
- 简单可靠，无需复杂状态机

### 6. LCD 显示策略

**决策**：检测到声音时**切换页面显示**，3 秒后返回

**流程**：
1. 检测声音 → 调用 `LCD_ShowSoundDetected()`
2. 清屏，显示"Sound Detected"
3. 启动 3 秒定时器
4. 超时后恢复原显示

**理由**：
- 用户需要视觉反馈确认触发成功
- 3 秒足够用户看到提示
- 自动恢复避免手动操作

## Risks / Trade-offs

### 风险 1：底噪阈值设置不当
- **风险**：阈值过低导致误触发，过高导致漏触发
- **缓解**：
  - 提供可调参数 `DELTA_THRESHOLD`
  - 添加串口调试输出实时信号幅度
  - 建议用户实测环境噪声后调整

### 风险 2：电源干扰导致误触发
- **风险**：MAX9814 对电源噪声敏感，可能产生虚假信号
- **缓解**：
  - 代码中添加数字滤波（滑动平均）
  - 建议使用 LDO 供电而非 DC-DC
  - 音频线使用屏蔽线

### 风险 3：DMA 中断响应延迟
- **风险**：DMA 中断处理耗时影响实时性
- **缓解**：
  - 中断中只做数据拷贝和标志设置
  - 复杂计算放到主循环
  - 使用高优先级 DMA 中断

### 风险 4：LCD 刷新影响音频采样
- **风险**：LCD 刷新占用 SPI 总线，可能影响 DMA
- **缓解**：
  - LCD 刷新和音频采样使用不同外设
  - ADC 使用 DMA 自动传输，不依赖 CPU
  - 测试最坏情况下的时序

### 风险 5：温度漂移影响
- **风险**：温度变化影响 ADC 参考电压和 MAX9814 偏置
- **缓解**：
  - 底噪动态跟踪可补偿缓慢漂移
  - 避免阈值设置过于敏感

### 风险 6：多声源同时触发
- **风险**：多人说话或背景噪声导致持续触发
- **缓解**：
  - 设置合理的冷却时间（500ms）
  - 阈值设置高于普通对话水平
  - 针对瞬态声音（响指）优化

## Open Questions

1. **最佳阈值参数**：需要通过实测确定 `DELTA_THRESHOLD` 的最佳值（建议初始 200）
2. **缓冲区最优大小**：128 点是否足够，是否需要调整到 256 点
3. **底噪跟踪速度**：滤波系数 0.98 是否合适，是否需要更快/更慢
4. **LCD 显示内容**：是否需要显示声音幅度条形图，还是仅文字提示
5. **多级别触发**：是否需要区分轻敲和重击（如 2 级阈值）
